Design Automation Pipeline

Concept-to-UI Agent Chain

Architecture plan for a composable 5-agent pipeline that takes any input (sketch, Figma mockup, PRD) through wireframing, implementation, and UX audit — tailored for the Osmos retail media platform.

Pipeline Overview

Five specialized agents, each armed with specific skills and tools. They run sequentially — each agent's output becomes the next agent's input. The orchestrator at the end ties the deliverables together into a single handoff package.

Agent Specifications

1 Interpreter Agent // input-normalizer

Purpose: Accept any input type — hand-drawn sketch photo, existing Figma URL, or text PRD — and produce a structured Screen Spec JSON that downstream agents can consume uniformly.

How it works:

For sketches: uses vision to identify UI regions, labels, navigation patterns, and data zones. Extracts layout hierarchy, component intent (e.g., "this looks like a data table with filters"), and flow direction.

For Figma URLs: calls use_figma to traverse the node tree, reads layer names, auto-layout settings, component instances, and text content. Maps existing design tokens to the Osmos system.

For PRDs / text briefs: parses requirements into screen zones using the design-skill's UX patterns, infers information architecture, and generates a layout spec.

Output: A structured JSON spec containing: page type, zone map (header, sidebar, content, footer), component intents per zone, data requirements, user flow connections, and any brand/domain constraints detected.

↓
2 Wireframer Agent // figma-builder

Purpose: Take the Screen Spec JSON and create a proper wireframe in Figma — low-fidelity first, then optionally promoted to hi-fi using Osmos design tokens.

How it works:

Creates a new Figma page (or file) via create_new_file / use_figma. Builds the frame hierarchy matching the zone map. Uses auto-layout for all containers. Places placeholder components (rectangles with labels for lo-fi, or searches for existing design system components via search_design_system for hi-fi).

Applies Osmos brand tokens: navy/gradient hero sections, #F2F2FC content backgrounds, 10px border-radius cards, Montserrat headings, Lato body text. Follows the layout rules from osmos-brand.

Output: A Figma file URL with named frames, proper auto-layout, and a component map (which Figma components were used where). Exports a PNG screenshot for the next agent.

Figma MCP (write) search_design_system osmos-brand design-skill
↓
3 Implementer Agent // react-coder

Purpose: Translate the wireframe + component map into production-ready React code using the @onlinesales-ai/ui design system. No custom HTML — every element maps to an existing Osmos component.

How it works:

Reads the component-index.md and UI-COMPONENT-REFERENCE.md. For each zone in the spec, selects the right component: Table for data grids, Card.Root for info blocks, Tabs.Root for navigation, Select/Combobox for dropdowns, etc. Uses dot notation for all compound components. Wraps in OsmosProvider with tokens.css import.

For sidebar/header/charts/filters — uses the package fallbacks (sidebar-v2, header-v2, highchart-v2, filter-form-v2). Style props use design tokens exclusively — no inline styles, no hardcoded hex values.

Uses configHocV2 wrapper for page components. Lazy loading for route registration. Follows the patterns in ROUTE-PAGE-GUIDE.md.

Output: React component files (.jsx), route config updates, and a standalone .html preview file for visual verification.

component-index.md @onlinesales-ai/ui ROUTE-PAGE-GUIDE.md osmos-brand (tokens)
↓
4 UX Auditor Agent // ux-strategist

Purpose: Run a structured UX audit on the implemented screen using Peter Morville's UX Honeycomb, tailored to Osmos's retail media domain. Produces a scored report with actionable fixes.

How it works:

Takes the Figma screenshot + React code + Screen Spec as input. Evaluates against 7 honeycomb dimensions, each with Osmos-specific criteria:

DimensionOsmos-Specific CriteriaWeight
UsefulDoes this screen directly serve a retail media workflow? (Campaign setup, yield optimization, reporting, onboarding)20%
UsableCan an ad ops manager complete the task in under 3 clicks? Are bulk actions supported?20%
DesirableDoes it feel like a premium SaaS tool? Brand consistency with Osmos design language?10%
FindableIs navigation predictable? Can the user locate this screen from the sidebar within 2 levels?15%
AccessibleWCAG 2.1 AA compliance, keyboard nav, 4.5:1 contrast, screen reader support15%
CredibleDoes data presentation inspire trust? Are metrics sourced, timestamped, and explainable?10%
ValuableDoes this screen contribute to breaking the M%G ceiling? Can we tie it to revenue uplift?10%

Also cross-references the design-skill's 99 UX guidelines and the osmos-pmm positioning to ensure the screen tells the right product story.

Output: A scored HTML audit report with per-dimension grades (A/B/C/D), specific fix recommendations, flow improvement suggestions, and a "business alignment score" that maps to Osmos's ceiling-break narrative.

design-skill (UX rules) osmos-pmm osmos-brand Vision (screenshot audit)
↓
5 Orchestrator Agent // tie-together

Purpose: Collect all outputs from agents 1–4 and produce a single handoff package. Optionally applies UX Auditor fixes automatically before packaging.

How it works:

Reads the audit report. If any dimension scores below B, it routes the specific fix back to the relevant agent (wireframe layout issue → Agent 2, component issue → Agent 3, brand issue → re-run with osmos-brand). After fixes, repackages everything.

Output: A handoff folder containing: Figma file link, React component code, UX audit report, Screen Spec JSON, and a summary brief with next steps.

What We Need From You

Here's everything we need to build and arm these agents. Organized by priority — "Must Have" items are blockers, "Should Have" make the agents smarter, "Nice to Have" are polish.

Must Have

Figma Access

  • Figma PAT (Personal Access Token) — with read+write scope. This powers Agents 1 and 2 to read existing designs and create new wireframes.
  • Figma plan key — the team/org key where new files should be created (get this from the Figma MCP whoami tool). We need to know which workspace the wireframes land in.
  • Design System file key — the file key for your Figma design system (the one at figma.com/design/58jL2Gbe53rBhxOysvHM82/Design-System-OS). Agent 2 uses search_design_system to pull components from this library.
Must Have

Component Library Context

  • component-index.md — the CLAUDE.md references this file at .claude/component-index.md but it wasn't found in the current folder. Agent 3 (Implementer) depends on this to map wireframe zones to real @onlinesales-ai/ui components. Please provide or point us to the right path.
  • UI-COMPONENT-REFERENCE.md — the full 113-component reference doc. Same dependency as above.
Must Have

Sample Inputs (at least 2-3)

  • 2-3 example inputs for testing — we need real examples of each input type you plan to use:
    • A hand-drawn sketch or whiteboard photo of a screen
    • A Figma URL of an existing mockup that needs implementation
    • A text PRD or brief describing a new screen
  • These become our first test cases for the skill-creator eval loop.
Must Have

Target Screen Types

  • Which screen types will this chain handle most often? Examples:
    • Campaign dashboards with charts + KPI cards
    • Settings/configuration pages with forms + tabs
    • Data tables with filters, bulk actions, and drill-down
    • Onboarding flows with step-by-step wizards
    • Reporting views with date pickers and export
  • Knowing the top 3-5 screen types lets us build specialized templates inside Agent 2 and 3 for much faster output.
Should Have

Existing Screen Examples

  • 2-3 "gold standard" screens — existing Figma designs or live pages in Localium Pulse that you consider well-designed. These become the quality benchmark for the UX Auditor and help calibrate Agent 2's layout decisions.
  • 1-2 "bad example" screens — screens with known UX issues. These help us calibrate the auditor's sensitivity and validate it catches real problems.
Should Have

Business Workflow Context

  • User persona descriptions — who uses Localium Pulse? (Ad ops managers, retail media directors, brand advertisers?) The UX Auditor's "Useful" and "Usable" scoring needs to know the primary user's mental model.
  • Core user workflows (top 3-5) — e.g., "Campaign creation flow: select inventory → set targeting → set budget → launch". These let Agent 4 evaluate whether a screen fits into the right workflow and catches flow breaks.
  • M%G context — any specific M%G benchmarks or ceiling-break metrics that the Auditor should reference when scoring the "Valuable" dimension.
Should Have

Routing & Architecture Docs

  • ROUTE-PAGE-GUIDE.md — the Agent 3 needs this to correctly register new components in the monorepo's routing system.
  • An example page component — a well-structured existing page (e.g., the Rooftop Onboarding page referenced in CLAUDE.md) so Agent 3 can match the exact patterns used in your codebase.
Nice to Have

Quality Refinements

  • UX audit rubric customization — are there specific UX pet peeves or non-negotiables for Osmos? (e.g., "every table must support CSV export", "no screen should have more than 2 scroll contexts"). These become hard rules in Agent 4.
  • Competitor screenshots — screenshots of Criteo, Kevel, Topsort, or Zitcha interfaces. The Auditor can use these as "what not to do" references and flag when an Osmos screen accidentally mirrors a competitor's pattern.
  • Accessibility requirements — beyond WCAG 2.1 AA, any specific requirements? (e.g., RTL support, specific screen reader testing targets, color blindness simulation requirements)

How We Build This

Phase 1: Foundation (Week 1)

Create the 5 skills using the skill-creator framework. Each agent becomes a skill with its own SKILL.md, reference files, and test cases. Start with Agent 1 (Interpreter) and Agent 2 (Wireframer) since they have the clearest input/output contract.

Deliverables: interpret-screen/SKILL.md, wireframe-builder/SKILL.md, 3 test cases each, eval benchmarks.

Phase 2: Implementation Engine (Week 2)

Build Agent 3 (Implementer). This is the heaviest agent — it needs deep knowledge of your 113 components, the monorepo structure, and the routing patterns. We'll embed the component-index as a reference file and build component-selection heuristics.

Deliverables: screen-implementer/SKILL.md, component mapping rules, 3 end-to-end tests (spec → code).

Phase 3: UX Intelligence (Week 2-3)

Build Agent 4 (UX Auditor) with the Osmos-specific honeycomb. Embed scoring rubrics, domain-specific heuristics, and the PMM positioning context. Calibrate against your gold-standard and bad-example screens.

Deliverables: ux-auditor/SKILL.md, scoring rubric reference file, audit report HTML template, calibration results.

Phase 4: Orchestration & Polish (Week 3)

Build Agent 5 (Orchestrator) and the fix-routing logic. Run the full chain end-to-end on 5+ real inputs. Tune based on results. Optimize skill descriptions for reliable triggering.

Deliverables: design-orchestrator/SKILL.md, full pipeline tests, description optimization via skill-creator's description improver.
